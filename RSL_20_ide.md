# Интегрированная среда разработки программ на языке RSL

### Создание

В этом режиме пользователю предоставляется возможность создать новый словарь для своих приложений.
Перед созданием система запрашивает имя нового словаря. Пользователю необходимо ввести любое имя с
расширением .def.
### Закрытие

При выборе этого режима происходит закрытие открытого словаря. При этом имя словаря, находившееся в
правом верхнем углу экрана, исчезает.
После того, как словарь был закрыт, пользователь не сможет работать ни в одном из режимов системы.
### Тестирование

При выборе пользователем данного режима производится проверка соответствия структур, описанных в
словаре, структурам физических файлов на диске.
В результате формируется протокол проверки (см. Рис. 1), в который заносятся сообщения системы об
обнаруженных несоответствиях.
Необходимо отметить, что эти сообщения не влияют на работу программного комплекса, поскольку вызваны
несоответствием размера поля в описании Btrieve-файла с размером ключа Btrieve-файла. Более подробная
информация о построении ключей Btrieve-файла содержится в документации, которая входит в комплект
поставки Pervasive SQL.
Рис. 1. Протокол тестирования словаря.
Система выдает следующие сообщения об ошибках:
 Файл не найден;
Структура файла определена в словаре, а файл с такой структурой и именем на диске не существует.
 Несоответствие размеров буферов (N1 – N2);
В скобках указывается размер записи в байтах в словаре (N1) и в файле (N2).
 Отсутствует сегмент (N1 – N2);
В скобках указываются номер ключа (N1) и номер сегмента (N2). Ключи и сегменты нумеруются, начиная с
нуля.
 Неверный размер или позиция сегмента N1-N2 (N3,N4-N5,N6);
Система указывает номер ключа (N1), номер сегмента (N2), позицию (N3) и размер (N4) в словаре, а также
позицию (N5) и размер (N6) в файле.
 Тип сегмента не соответствует типу поля (N1 – N2);
В скобках указываются номер ключа (N1) и номер сегмента (N2). Ключи и сегменты нумеруются, начиная с
нуля.
Работа со стандартными формами отчетов подробно описана в Руководстве "Общие документы. Приложение".

### Конвертирование в DDF

Система позволяет конвертировать словарь в формат стандартного словаря фирмы Novell. Это позволяет
использовать подготовленную базу данных пользователя в приложениях, разработанных для программ третьих
фирм, например, Microsoft Access.
В процессе конвертирования в рабочем каталоге создается пустой словарь, состоящий из трех файлов: file.ddf,
index.ddf и field.ddf. В этот словарь заносится вся информация о базе данных из открытого DEF-словаря.
Дозапись в DDF
При выборе данного режима в словарь формата DDF, находящийся в рабочем каталоге, дописываются
недостающие данные о файлах открытого DEF-словаря.
В результате дозаписи формируется протокол, в который система заносит сообщения о возникших ошибках
(см. Рис. 2). Работа со стандартными формами отчетов подробно описана в Руководстве "Общие документы.
Приложение".

## Директории

С каждым словарем связан файл конфигурации. Файл конфигурации создается в том же каталоге, что и
словарь. Его имя состоит из имени файла словаря, к которому добавляется символ "$", и расширения .cfp. В
файле конфигурации хранится вся информация, которую пользователь вводит в режиме "Директории".
На Рис. 3 представлен стандартный файл конфигурации, значения в котором установлены по умолчанию.
В полях, определяющих директории файлов, пользователь вводит список директорий, разделенных символом
";".
При обращении к файлам в интерактивном режиме или из макропрограммы система выполняет поиск в
установленных директориях.
При работе с Btrieve-файлами в интерактивном режиме поиск файлов выполняется только в первом каталоге из
этого списка. Если обращение к файлам производится из макропрограммы на языке RSL, то поиск файлов
выполняется во всех каталогах из списка.

В поле ВНЕШНИЙ ТЕКСТОВЫЙ РЕДАКТОР пользователь может ввести имя редактора, который будет
использован при работе с макропроцедурами (см. раздел "Список макропроцедур" на стр. 110).

диске не изменяется.

Рис. 4. Список структур файлов словаря.
Поля заполняются непосредственно в списке.
В поле ИМЯ ФАЙЛА выводится имя Btrieve файла.
### Поле ВЛАДЕЛЕЦ содержит имя владельца файла (пароль).

В поле ФФ пользователь при помощи встроенного списка вводит атрибуты установки пароля файла. Ввод
данных в это поле осуществляется только при установке имени владельца файла.
В списке атрибутов определены следующие значения:
 Encryption – содержимое файла будет зашифровано.
 Read-Only mode – если указан этот атрибут, то доступ к файлу только для чтения осуществляется без
указания пароля. Если атрибут не указан, то пароль должен быть указан для любого доступа к файлу.
В поле СТР. вводится размер страницы Btrieve-файла. Это значение должно быть кратным 512 и не более 4096.
В поле ФЛАГИ при помощи встроенного списка устанавливаются флаги Btrieve-файла. Список флагов
содержит следующие значения:
 Variable length – при установке данного флага Btrieve-файл будет иметь записи переменой длины;
 Truncate blanks – при установке этого флага будет осуществляться подавление концевых пробелов;
 Preallocate – этот флаг не поддерживается в данной версии языка RSL;
 Compress data – при установка этого флага система выполняет сжатие данных в Btrieve-файле;
 Key-only file – при установке данного флага Btrieve-файл будет создан как файл только ключей. Этот флаг
устанавливается только в случае, если файл имеет только один ключ и маленькую структуру. Без
необходимости не следует создавать такие файлы, так как на их использование накладываются
ограничения;
 10% free space, 20% free space, 30% free space – эти три флага используются совместно с флагом Variable
Length. Количество процентов означает, сколько процентов страницы Btrieve-файла будет зарезервировано
для расширения записи переменной длины.
Все флаги являются стандартными и подробно описаны в документации к Btrieve.
Поле МДПЧ используется в совокупности с полем В и заполняется в том случае, если в поле ФЛАГИ
установлен флаг Variable length.
В поле В можно установить отметку с помощью клавиши [Space]. В зависимости от ее наличия заполняется
поле МДПЧ:
 отметка в поле В не установлена − в этом случае в поле МДПЧ задается максимальный размер в байтах
переменной части записи файла;
 отметка в поле В установлена − в этом случае переменная часть записи файла будет рассматриваться как
объект типа BLOB, и в поле МДПЧ задается размер буфера, в который считываются данные из поля BLOB.
В поле ПРИМЕЧАНИЕ пользователь вводит любую текстовую информацию о файле.

Поле ФЛАГИ содержит набор флагов ключей Btrieve. Наиболее важные из них следующие:
 Duplicates – значение ключа может дублироваться. Если этот флаг указан для одного сегмента ключа, то он
должен быть указан и для всех остальных сегментов данного ключа.
 Modifiable – значение ключа может быть модифицировано.
 Extended type – этот флаг указывает на то, что данный сегмент ключа принадлежит к расширенным типам
ключей Btrieve. Этот ключ следует указывать для всех сегментов!
 Segment – данный флаг указывает на то, что следующий сегмент принадлежит этому же ключу.
 Alternate order – данный флаг можно указать для полей типа STRING или SNR. Если флаг установлен, то
при сортировке символы верхнего и нижнего регистра в этих полях будут считаться одинаковыми.

### Трансляция макрофайла

В интерпретаторе RSL предусмотрена возможность генерировать прекомпилированные бинарные файлы (пи-
код) для ускорения процедуры синтаксического разбора макропрограммы.
# Интегрированная среда разработки программ на языке RSL

При использовании пи-кода интерпретатор RSL освобождается от фазы синтаксического разбора, что
сокращает время, необходимое для выполнения макрозадания.
Пи-код сохраняется в файле с расширением *.RSM. Имя этого файла и его расположение на диске те же, что и у
соответствующего макрофайла.
Сгенерировать пи-код можно, выбрав соответствующий пункт меню консольного приложения
"Процедуры\Трансляция макро-файла", или в режиме отладки, воспользовавшись клавишами [Alt+F9].
Если в результате трансляции макрофайла будет обнаружено, что на диске уже есть соответствующий rsm-
файл, и он актуален (то есть имеет дату более позднюю, чем макрофайл), то будет выведено соответствующее
сообщение, и трансляция будет отложена. Если после этого в режиме отладки внести изменения в макрос и
запустить его на выполнение, то будет выдано сообщение о неактуальности существующего rsm-файла.

одновременно с нажатой клавишей [Shift].

Работать с диалогами можно как при помощи клавиш перемещения курсора, так и с помощью мыши. Если
пользователь работает с мышью, то процесс изменения формы диалога значительно упрощается. Чтобы
изменить местоположение элемента диалога с помощью мыши, необходимо установить на него курсор и
перемещать мышь с одновременно нажатой правой кнопкой. Чтобы переместить диалоговое окно, необходимо
зафиксировать курсор на верхней границе диалогового окна и перемещать мышь с нажатой правой кнопкой.
Для изменения размеров окна нужно зафиксировать курсор в правом нижнем углу окна и перемещать мышь с
нажатой правой кнопкой.
