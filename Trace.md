# Trace

## Введение

Внимание!
Данное руководство используется только при работе ИБС RS-Bank V.6
на СУБД Oracle.
В ИБС RS-Bank V.6 предусмотрена возможность сбора детальной информации об
изменениях в системе, произошедших в результате работы пользователя. С этой целью
используется механизм трассировки. В отличие от журнализации, при которой ведется
учет прикладных действий пользователя (например, изменений в реестре настроек банка),
выполнение 
трассировки 
позволяет 
рассмотреть 
техническую 
сторону 
работы
пользователя в системе (например, проследить, какие таблицы базы данных были
задействованы при работе).
Настоящее Руководство содержит описание механизма трассировки, необходимость
использования которого возникает при обнаружении ошибок в работе системы. Применение
механизма трассировки позволяет не только определить саму ошибку, но и получить
дополнительную информацию о ней (вплоть до шага, на котором она возникла), что важно
для выявления и устранения ее причин.
При ссылке на информацию, расположенную в настоящем руководстве, указывается номер
страницы (в скругленном прямоугольнике 
), соответствующий нужному Вам разделу, или
гиперссылка. Ссылка на раздел другого руководства дается с указанием названия главы и
раздела, содержащего требуемую информацию.

## Общие положения

Трассировка может быть запущена из любого режима любой подсистемы, что позволяет
значительно сузить интервал поиска возникшей ошибки.
Предусмотрено два вида трассировки:
- трассировка RS-Trace – трассировка действий, выполняемых в ИБС RS-Bank V.6;
- трассировка Oracle – трассировка действий сторонних приложений, использующих СУБД
Oracle.
Включение и выключение трассировки выполняется в специальном режиме
. Кроме этого,
предусмотрена возможность запуска трассировки безынтерфейсно с использованием
процедур языка RSL (см. раздел "Модуль BankInter" Руководства "Интерфейсы языка RSL
для взаимодействия с ИБС RS-Bank V.6. Часть 1" (файл Books\Tools\CoreRSLprc_1.pdf)):
- SetRSTrace – процедура предназначена для включения/выключения трассировки RS-
Trace.
- SetOraTrace – процедура предназначена для включения/выключения трассировки Oracle.
Также с использованием процедуры языка RSL (см. раздел "Модуль PaymInter" Руководства
"Интерфейсы языка RSL для взаимодействия с ИБС RS-Bank V.6. Часть 3" (файл
Books\Tools\CoreRSLprc_3.pdf)) Bnk_ToRSTrace можно выполнять выгрузку сообщений в
штатный лог-файл RS-Trace.
Для трассировки RS-Trace предварительно необходимо выполнить настройку
. По итогам
трассировки RS-Trace формируются файлы трассировки (файлы с расширением .log) с
информацией о системных изменениях. Все изменения, произошедшие в системе в
результате работы пользователя (например, изменение значений в полях таблицы базы
данных), называются событиями. Информация об указанных событиях представлена в
файлах трассировки в виде таблицы. Для оптимизации анализа полученной информации
все события сгруппированы по источникам, затем по категориям и уровням
детализации. При этом каждый источник трассировки используется для сбора информации
об определенном наборе событий.
Таблица в файле трассировки содержит следующие столбцы:
- source – наименование источника трассировки;
- category – наименование категории и события;
- detail – уровень детализации информации;
- time – время возникновения события;
- file name – наименование системного файла, в котором выполнены изменения, и номер
измененной строки в нем;
- message – описание произошедшего события.
Перечень источников, категорий и событий, используемых для трассировки RS-Trace,
приведен в Приложении
.
Перед трассировкой Oracle необходимо в файле oratrace.ini задать следующие параметры
настройки запуска трассировки Oracle:
- Event – номер события. Предусмотрена возможность задания нескольких номеров
событий.
- Level – уровень трассировки события. Трассировка события 10046 выполняется всегда,
при этом параметр Level имеет значение "12".
Пример.
Event = 10200
Level = 1
Файлы трассировки создаются автоматически при включении трассировки и сохраняются на
сервере в каталоге ..\UDump.

По умолчанию таблица в файле трассировки содержит все перечисленные столбцы в

Использование механизма трассировки позволяет контролировать события, возникающие
при использовании данных следующих типов: dmoney, decimal, double, longdouble. С этой
целью в файле rstrace.ini необходимо задать раздел [TRACESOURCE] = ToolsTrace и
указать категорию [CATEGORY] = NewMoney.
Далее следует задать события, для которых требуется трассировка. Возможные события:
- Div100 – событие первого уровня детализации. Выводится при выполнении операции
деления величин, имеющих тип dmoney, decimal, double или longdouble, на 100 средствами
языка RSL.

## Трассировка изменения

значения
переменной CNum
Переменная CNum представляет собой уникальный идентификатор сессии пользователя.
Для сбора информации об изменении значения переменной CNum в файле rstrace.ini
необходимо задать раздел [TRACESOURCE] = ToolsTrace, указать категорию [CATEGORY] =
CNUM и событие Init с четвертым уровнем детализации. В результате выполнения
указанных действий в файл трассировки будет занесено новое значение переменной CNum.

## Трассировка возникновения

критичных
ошибок
Механизм трассировки позволяет вести учет обращений к процедурам, вызываемым при
возникновении критичных ошибок. Указанные ошибки приводят к аварийному выходу из
подсистемы. Для запуска указанной трассировки необходимо в файле rstrace.ini задать
раздел [TRACESOURCE] = ToolsTrace и указать категорию [CATEGORY] = Exception.
Далее, в зависимости от требуемых ошибок, необходимо указать следующие события:
- Для критичных ошибок, связанных с функционированием операционной системы и
установленного оборудования (т.н. необработанных структурных исключений) – событие
Fatal с четвертым уровнем детализации. При этом выполняется учет вызовов процедуры
FatalHandler.
- Для критичных ошибок подсистемы, предусмотренных списком исключений (т.н.
необработанных исключений С++) и сопровождающихся информацией об ошибке –
событие Terminate с четвертым уровнем детализации. В этом случае выполняется учет
вызовов процедуры TerminateHandle.
- Для критичных ошибок подсистемы, которые не предусмотрены списком исключений и не
сопровождаются информацией об ошибке (т.н. unexpected-исключений С++) – событие
Unexpected с четвертым уровнем детализации. В этом случае выполняется учет вызовов
процедуры UnexpectedHandler.
- Для критичных ошибок, возникших при выполнении операций с данными типа decimal (т.н.
исключений FDecimal) – событие FDecimal с четвертым уровнем детализации. При этом
выполняется учет вызовов процедуры FDecimal_Error.
Трассировка включения/выключения
трассировки
В системе предусмотрена возможность выполнить трассировку включения/выключения
самого механизма трассировки. С этой целью в файле rstrace.ini необходимо задать раздел
[TRACESOURCE] = ToolsTrace, указать категорию [CATEGORY] = Trace и событие
EnableTrace с четвертым уровнем детализации. В результате в файл трассировки будет
выведено сообщение "TraceOn" или "TraceOff".

если не могут быть открыты все файлы трассировки (например, в файле rstrace.ini

трассировки и его наименование.
