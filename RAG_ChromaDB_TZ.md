# Техническое задание на разработку системы RAG на базе ChromaDB с административным интерфейсом

## 1. Цель и область применения

**Цель:** создать централизованную векторную базу знаний для команды разработчиков, работающих с редким языком программирования. Система должна обеспечивать семантический поиск по документации (600+ страниц) и примерам кода (10 000+ строк), интеграцию с Kimi Code через MCP и простой веб-интерфейс для мониторинга состояния и управления обновлениями.

**Область применения:** локальная сеть команды, развёртывание на выделенном сервере/NAS в Docker.

---

## 2. Архитектура системы

Система состоит из четырёх логических компонентов:

1. **Векторная база данных (ChromaDB Server)** — центральное хранилище эмбеддингов и метаданных. Доступ по HTTP API.
2. **Сервис индексации (Indexer)** — компонент, выполняющий чтение исходных файлов из Git-репозитория, разбиение на чанки, вычисление эмбеддингов и загрузку в ChromaDB.
3. **MCP-шлюз (MCP Gateway)** — лёгкий сервер для подключения Kimi Code разработчиков. Предоставляет только инструменты чтения (поиск, получение фрагментов).
4. **Административный интерфейс (Admin UI)** — веб-приложение для просмотра состояния системы и ручного запуска переиндексации.

---

## 3. Требования к данным

### 3.1. Источники данных
- **Документация:** текстовые файлы в форматах `.md`, `.txt`, `.rst`, `.mdx`.
- **Примеры кода:** файлы с расширением `.mac` (Maxima/Assembler), а также иные текстовые исходники.
- **Хранение исходников:** Git-репозиторий, клон которого размещается на сервере через Docker volume.

### 3.2. Кодировки
- Все текстовые файлы должны быть в кодировке **UTF-8**.
- Система должна корректно обрабатывать файлы с кириллицей, математическими символами и спецсимволами.
- При обнаружении файлов в иных кодировках (Windows-1251, CP866, KOI8-R) индексатор должен выполнять транслитерацию/перекодирование в UTF-8 перед обработкой.

### 3.3. Чанкинг (разбиение на фрагменты)
- **Для документации:** разбиение по логическим заголовкам (Markdown-заголовки) с перекрытием. При отсутствии явной структуры — по объёму (примерно 1000–1500 символов) с перекрытием 200 символов.
- **Для кода (.mac):** разбиение по логическим границам — определения функций, операторов, блоков. Не допускается разрезание функций посередине.
- **Метаданные каждого чанка:** `source` (путь к файлу), `doc_type` (`docs` или `code_examples`), `language`, `chunk_index`, `file_ext`, `version` (при наличии).

### 3.4. Коллекции
- `lang_docs` — основная коллекция для документации и примеров.
- Опционально: `kimi_memory` — отдельная коллекция для рабочих заметок Kimi (если требуется функция памяти), с изолированным доступом.

## 4. Функциональные требования

### 4.1. ChromaDB Server
- Развёртывание в Docker с персистентным хранилищем (volume).
- Постоянное хранение данных между перезапусками контейнера.
- Базовая аутентификация по статическому токену (Bearer token).
- HTTP API на порту 8000.
- Ограничение использования RAM в пределах 4 ГБ (конфигурация Docker).

### 4.2. Сервис индексации (Indexer)
- **Ручной запуск:** через административный интерфейс (кнопка «Обновить из Git»).
- **Автоматический запуск:** опционально по расписанию (cron) или через CI/CD (webhook от Git).
- **Процедура обновления:**
  1. Выполнение `git pull` из заданного репозитория и ветки.
  2. Очистка устаревших чанков (или полная пересоздание коллекции).
  3. Разбиение файлов на чанки согласно правилам п. 3.3.
  4. Вычисление эмбеддингов (модель `all-MiniLM-L6-v2` или аналогичная, 384-мерные векторы).
  5. Загрузка в ChromaDB с метаданными.
- **Логирование:** фиксация времени запуска, количества обработанных файлов, количества чанков, ошибок чтения/кодировок.
- **Идемпотентность:** повторный запуск не должен создавать дубли.

### 4.3. MCP-шлюз
- Подключение к центральному ChromaDB по HTTP (не stdio).
- **Инструменты, доступные Kimi Code:**
  - `search_docs` — семантический поиск по коллекции `lang_docs` с фильтрацией по `doc_type`.
  - `search_api_reference` — поиск определений функций/типов (с приоритетом точных совпадений).
  - `get_example_code` — поиск примеров кода по теме.
- **Доступ:** только чтение. Инструменты записи, удаления и модификации коллекций **исключены**.
- Транспорт: stdio (локальный процесс на машине разработчика), который проксирует запросы на сервер ChromaDB.
- Конфигурация через переменные окружения: `CHROMA_HOST`, `CHROMA_TOKEN`.

### 4.4. Административный интерфейс (Admin UI)
- **Статус системы (дашборд):**
  - Индикатор доступности ChromaDB (heartbeat).
  - Количество документов (чанков) в коллекции `lang_docs`.
  - Дата и время последней успешной индексации.
  - Версия Git-коммита, на которой построен текущий индекс.
  - Список файловых источников в индексе.
- **Управление:**
  - Кнопка «Pull & Reindex» — запуск полного цикла обновления из Git с отображением прогресса (лог в реальном времени).
  - Возможность прервать процесс индексации.
- **Просмотр данных:**
  - Список коллекций.
  - Просмотр чанков (постранично) с фильтрацией по `doc_type` и `source`.
  - Тестовый семантический поиск (ручной ввод запроса, вывод топ-5 результатов).
- **Логи:**
  - История запусков индексации (время, статус, количество файлов/чанков, ошибки).
  - Логи запросов от MCP-шлюза (опционально, для аудита).

## 5. Нефункциональные требования

- **Производительность:** время ответа на семантический запрос — не более 2 секунд для коллекции до 100 000 чанков.
- **Масштабируемость:** архитектура должна позволять в будущем перейти на более мощные эмбеддинг-модели (например, `BGE-M3`) без изменения структуры хранения.
- **Отказоустойчивость:** при падении ChromaDB MCP-шлюз и Admin UI должны корректно сообщать о недоступности базы.
- **Резервное копирование:** персистентный volume ChromaDB должен включаться в систему бэкапов сервера (snapshot каталога данных).
- **Обновления:** процедура обновления Docker-образов ChromaDB и компонентов системы не должна приводить к потере индекса.

---

## 6. Безопасность и доступ

- **ChromaDB:** доступ по токену (Bearer). Токен хранится в переменных окружения Docker, не в коде.
- **MCP-шлюз:** разработчики получают только read-only токен/конфигурацию. Нет доступа к Admin UI.
- **Admin UI:** доступ по статическому API-ключу (токену), передаваемому в заголовке или параметре запроса. SSO не требуется. Доступ ограничен администраторами.
- **Сетевая изоляция:** порт ChromaDB (8000) доступен только внутри локальной сети (или Docker network), не выставлен наружу.
- **Git-репозиторий:** доступ к исходникам через SSH-ключ или deploy-token, хранящийся на сервере.

---

## 7. Интеграция с Kimi Code

- Разработчики подключают MCP-шлюз через команду `kimi mcp add` с указанием пути к локальному клиентскому скрипту.
- Kimi Code получает инструменты поиска и использует их автоматически при запросах, связанных с языком программирования.
- Документация не загружается в контекст чата целиком — только релевантные фрагменты, полученные через MCP.

---

## 8. Этапы разработки и приёмка

| Этап | Содержание | Критерий приёмки |
|------|-----------|------------------|
| **1. Инфраструктура** | Docker Compose с ChromaDB, volume, токен-аутентификация. | ChromaDB отвечает на `/api/v1/heartbeat`, данные сохраняются после перезапуска. |
| **2. Индексация** | Скрипт чанкинга и загрузки `.md` и `.mac` файлов из Git. | 600+ страниц и 10 000+ строк кода загружены в коллекцию `lang_docs`, метаданные корректны. |
| **3. MCP-шлюз** | Read-only MCP-сервер с тремя инструментами. | Kimi Code успешно вызывает поиск и получает релевантные фрагменты. |
| **4. Admin UI** | Дашборд статуса, кнопка обновления, просмотр чанков, тестовый поиск. | Администратор видит количество документов, запускает reindex, видит лог, выполняет тестовый поиск. |
| **5. Тестирование** | Нагрузочное тестирование, проверка кодировок, проверка чанкинга кода. | Нет кракозябр в индексе, функции `.mac` не разрезаны, поиск по API возвращает корректные сигнатуры. |
