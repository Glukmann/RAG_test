## тестирование

используется 
для

видов 
тестирования
программного обеспечения, направленных на выявление ошибок в уже протестированных
блоках исходного кода программы. Ошибки, приводящие к ситуациям, когда после внесения
изменений система перестает работать, называются регрессионными ошибками. Такие
ошибки часто возникают при работе с развивающимися программными продуктами. Главной
целью регрессионного тестирования является снижение вероятности возникновения ошибок
при вводе новой версии программного продукта в промышленную эксплуатацию.
Регрессионное тестирование подразумевает тестирование всех функций системы при
каждом внесении в нее изменений. В случае ручного тестирования этот процесс
предполагает огромные временные затраты и трудозатраты. Помимо этого сотрудникам
компании, занимающейся разработкой программного обеспечения, часто приходится
выполнять диагностику причин возникновения ошибок в версиях программного продукта,
переданных клиентам. Процесс написания пользователями алгоритма воспроизведения
ошибки также требует от них значительных трудозатрат.
Для решения задачи организации регрессионного тестирования, а также проблемы поиска и
локализации ошибок, возникающих в программных продуктах RS-Bank v.6.20 и v.5.50
компании R-Style Softlab, реализован универсальный инструмент записи и воспроизведения
событий (RS-Application Testing Machine или RS-ATM).
Данное руководство содержит описание инструмента записи и воспроизведения событий
RS-ATM, используемого в программном продукте RS-Bank v.6.20.

Инструмент записи и воспроизведения
событий (RS-ATM)
Основными целями инструмента записи и воспроизведения событий (RS-ATM) являются:
- Автоматизация процесса обнаружения ошибок, возникающих в системе RS-Bank V.6, и
диагностика причин их возникновения.
- Минимизация временных затрат клиентов на написание алгоритма воспроизведения
ошибки.
- Создание встроенной, высокопроизводительной системы тестирования.
Инструмент RS-ATM предназначен для:
- Передачи информации о последовательности действий, которая привела к возникновению
ошибки в системе, от клиента к клиентским службам.

ИБС. В случае возникновения ошибок в системе этот файл, предварительно

- Тестирования системы RS-Bank V.6.
Инструмент RS-ATM применяется для создания набора файлов, содержащих описание
действий, выполняемых в системе. При регрессионном тестировании в процессе
диагностики причин возникновения ошибок в системе воспроизведение этих файлов
может сопровождаться выполнением дополнительных действий, которые могут приводить

Помимо этого существует возможность одновременного, параллельного воспроизведения
нескольких файлов-скриптов на одном рабочем месте, что значительно ускоряет процесс
тестирования.

Настройка параметров работы
инструмента записи и воспроизведения
событий (RS-ATM)
На порядок работы инструмента записи и воспроизведения событий влияет следующая
группа настроек реестра банка подсистемы "Главная книга бэк-офисов" COMMON \ ATM:
- ВОСПРОИЗВЕДЕНИЕ \ SCRIPTDIR – каталог, из которого запускаются скрипты. По
умолчанию имеет значение "ATM".
- ВОСПРОИЗВЕДЕНИЕ \ РАЗРЕШЕНИЯ МОДУЛЯ – содержит список пользователей,
которым разрешено воспроизведение событий. Номера пользователей указываются через
запятую. По умолчанию настройка содержит значение * – воспроизведение событий
разрешено всем пользователям.
- ЗАПИСЬ \ LOGDIR – каталог, в который производится запись файлов. По умолчанию
имеет значение "Log_ATM".
- ЗАПИСЬ \ WRITE_TIME – определяет необходимость записи штампа времени в файлы,
создаваемые при записи или воспроизведении скрипта. Настройка может содержать
следующие значения:
- YES – запись штампа времени разрешена.
- NO – запись штампа времени не разрешена.
- ЗАПИСЬ \ РАЗРЕШЕНИЯ МОДУЛЯ – содержит список пользователей, которым разрешена
запись событий. Номера пользователей указываются через запятую. По умолчанию
настройка содержит значение * – запись событий разрешена всем пользователям.

## Технологическая модель

использования
Управление системой RS-Bank V.6 основано на использовании механизма сообщений.
Каждое событие, происходящее в системе: интерфейсное (появление формы на экране,
ввод данных в графу, нажатие клавиши) или прикладное (компиляция макрофайла, запуск
макроса на выполнение) – генерирует сообщение. Инструмент RS-ATM отслеживает эти
сообщения и может изменять их, управляя, таким образом, состоянием системы RS-
Bank V.6.
Внимание!
Работа с инструментом RS-ATM возможна только при наличии
лицензии на запись/воспроизведение событий.
Выполнять 
запись 
и 
воспроизведение 
событий 
могут 
только
пользователи, 
указанные 
в 
настройках 
банка
COMMON \ ATM \ ЗАПИСЬ \ РАЗРЕШЕНИЯ 
МОДУЛЯ
 
и
COMMON \ ATM \ ВОСПРОИЗВЕДЕНИЕ \ РАЗРЕШЕНИЯ МОДУЛЯ
.
Инструмент RS-ATM фиксирует события, происходящие в системе RS-Bank V.6, в виде
файлов скриптов. В дальнейшем эти файлы скриптов используются инструментом RS-ATM
для эмуляции работы пользователя. Скрипт представляет собой файл, содержащий
последовательность событий, записанную посредством специальных команд и элементов
языка программирования
.
Внимание!
Инструмент RS-ATM позволяет записывать события, возникающие при
работе с мышью, но не поддерживает возможность воспроизведения
таких событий.
Выделяют 3 режима работы инструмента записи и воспроизведения действий пользователя:
- Режим записи. В этом режиме инструмент RS-ATM фиксирует события, происходящие в
ИБС RS-Bank V.6, и кодирует их в команды с определенными атрибутами. Команды
записываются 
в 
файл 
скрипта 
в 
последовательности, 
отражающей 
появление
соответствующих им событий.
- Режим воспроизведения. В режиме воспроизведения инструмент RS-ATM выполняет
чтение команд записанного ранее файла скрипта, декодирует эти команды и с помощью
механизма сообщений выполняет эмуляцию работы пользователя в системе. Перед
воспроизведением файла скрипта автоматически выполняется компиляция этого файла,
что позволяет обнаружить все синтаксические ошибки, а также ошибки обращения к
неизвестным командам/переменным/макрофункциям еще на этапе компиляции.
- Совместный режим. При работе в совместном режиме воспроизведение уже созданного
файла-скрипта сопровождается записью нового файла.
Перед стартом системы RS-Bank V.6 для запуска инструмента в одном из перечисленных
выше режимов необходимо в командной строке указать один из следующих параметров:
- /rec:<имя файла для записи> – для запуска инструмента в режиме записи. Если файл,
имя которого указано после параметра /rec, уже существует, то после запуска инструмента
в режиме записи содержимое этого файла будет перезаписано.
- /play:<имя файла для воспроизведения> – для работы инструмента в режиме
воспроизведения.
- /xmlplay:<имя файла для воспроизведения> – для воспроизведения скриптов в
формате XML. Формат XML является устаревшим. Ключ xmlplay используется только для
совместимости со скриптами, записанными в старом формате.
Внимание!
Сочетание 
параметров 
/play 
и 
/xmlplay

недопустимо.
Если необходимо запустить инструмент в совместном режиме, то в командной строке
последовательно указываются параметры /rec и /play (или /rec и /xmlplay).
Помимо указанных выше параметров инструмент RS-ATM допускает возможность
использования параметра /hide_msg, который позволяет скрыть вывод сообщений об
ошибках воспроизведения/компиляции на экран.
Запись скриптов по умолчанию производится в каталог …\ Obj\ Log_ATM, воспроизведение
скриптов выполняется из каталога …\ Obj\ ATM. Для указания других каталогов можно
использовать настройки банка COMMON \ ATM \ ВОСПРОИЗВЕДЕНИЕ \ SCRIPTDIR
 и
COMMON \ ATM \ ЗАПИСЬ \ LOGDIR

Примечание.
Если в настройке COMMON \ ATM \ ВОСПРОИЗВЕДЕНИЕ \ SCRIPTDIR

происходит поиск файла.
При указании в командной строке параметра /play автоматически выполняется запись логов
воспроизведения. Логи – это файлы, имеющие следующие расширения:
- .log – файл лога воспроизведения, содержащий только события системы RS-Bank V.6, а
также информацию в виде комментариев, которая помогает пользователю выполнять
отладку скрипта.
- .err – файл, содержащий ошибки, которые возникали в процессе компиляции скрипта или
его воспроизведения.
- .inf – файл, предназначенный для отображения информации о ходе выполнения скрипта.
Содержимое этого файла определяется с помощью команд
 Log и LogAndInf.
Информация о ходе выполнения скрипта и возникающих ошибках содержится в лог-файлах
и имеет следующий вид //AAA-XXXXX>, где:
- AAA – вид сообщения. Может принимать следующие значения:
- sys – системное сообщение;
- err – сообщение об ошибке;
- wrn – сообщение об ошибке;
- inf – информационное сообщение, значение параметра команды Inf;
- log – информационное сообщение, значение параметра команды Log.
- XXXXX – номер сообщения.
После символа ">" содержится текст сообщения.
Примечание.
В файл скрипта и лог-файлы можно записывать точное время выполнения
каждого события. Возможность записи штампа времени в лог-файлы
регулируется настройкой банка COMMON \ ATM \ ЗАПИСЬ \ WRITE_TIME

## Виды событий

С точки зрения управления ИБС RS-Bank V.6, в инструменте RS-ATM различают следующие

## виды событий

:
- Событие-эмуляция.
Событие-эмуляция – это событие, изменяющее состояние ИБС RS-Bank V.6. К таким
событиям относятся:
- Ввод данных в поле и нажатие клавиши – эти события фиксируются с помощью команд
ProcField и Keys соответственно.
- Выбор горизонтального/вертикального пункта меню – команда MenuItem.
- Реакция на окно редактирования файла – команда EditFile.
- Ответ на окно сообщения (выбор кнопки в окне сообщения, либо нажатие клавиши на
окне сообщения) – команда MsgWindow.
- Событие-маркер.
Событие-маркер используется для осуществления контроля текущего состояния ИБС RS-
Bank V.6 и местоположения воспроизведения. К таким событиям относятся:
- Появление панели/скроллинга на экране – это событие фиксируется с помощью
команды PanelOn.
- Удаление панели/скроллинга с экрана – PanelOff.
- Любое событие-эмуляция, для которого указаны данные контроля. Например, для
события эмуляции нажатия клавиши в качестве таких данных может выступать текст
поля в момент входа в это поле, а для события эмуляции ответа на окно сообщения это
может быть текст сообщения, а также название выбранной кнопки.

- Возможность вызова RSL-макросов. При этом возможно импортировать модуль rsatm,
который реализует команды записи/чтения переменных, доступных в скрипте в точке
вызова RSL-макроса
.
- Возможность использования системных переменных
, с помощью которых можно

- Управление трассировкой

Использование переменных
В скриптах, созданных с помощью инструмента RS-ATM, можно использовать переменные
следующих типов:
- Integer – целочисленный знаковый тип (числа от - 2147483648 до + 2147483648).
- Boolean – логический тип (значение true или false).
- String – строковый тип.
- Timer – переменные-таймеры.
Примеры объявления переменных
//Создание переменных типа variant:
var
a,b,c;
//Создание типизированных переменных:
var

//Создание типизированных переменных с инициализацией их
значений:
var
f:boolean=false, g:string="some text";
Переменные любых типов могут быть преобразованы к типу string.
Пример преобразования переменных к типу string
var
a:integer=1024,
b:boolean=false,
c:string="some text";
c=c+"; a= "+a+"; b= "+b;
//В результате выполнения указанного кода переменная "с"
будет иметь значение "some text; a= 1024; b= false".
Переменные-таймеры обладают следующими особенностями:
- для сброса переменной в нуль необходимо установить ей любое значение;
- для того чтобы определить значение переменной, необходимо обратиться к ней.
Пример работы с переменной типа timer
var

//Через 5 секунд работы скрипта после того, как была

объявлена переменная t

равна 0
//Через 50 секунд работы скрипта после того, как была
объявлена переменная t, и 35 секунд с момента ее повторной

Использование операторов if и while
Условный оператор if в инструменте RS-АТМ имеет упрощенную реализацию и не допускает
использование логических операндов типа "ИЛИ", "И" и т.п.
Синтаксис использования оператора if:
if (<выражение 1><знак сравнения><выражение2>)
…
end;
В качестве знака сравнения может использоваться один из следующих: >, <, >=, <=, ==, !=.
Сравниваемые выражения должны иметь одинаковый тип.
Пример использования условного оператора if
var
a:integer=10, b:integer=10
if (a+b>b+a)
…
end;
Оператор цикла while имеет ту же реализацию, те же ограничения и тот же синтаксис
использования, что и условный оператор if. Допустимы циклы любой степени вложенности.
Синтаксис использования оператора while:
while (<выражение 1><знак сравнения><выражение2>)
…
end;
Пример использования оператора цикла while
var
i:integer=10;
while (i > 2)
i = i - 1;
LogAndInf("Значение переменной i: "+i);
end;
Увеличение/уменьшение счетчика, используемого в цикле, выполняется операциями
сложения/вычитания.
При использовании цикла можно применять команду continue. При этом будут пропущены
все команды, следующие за ней, и цикл продолжится со следующей итерации.
Для прерывания цикла предназначена команда break. Команда break прерывает тот цикл, в
котором она находится, не затрагивая внешние циклы; при этом управление передается на
первый элемент после "end", относящийся к данному циклу.
Использование механизма подписок
К 
специальным 
функциональным 
возможностям 
утилиты 
тестирования 
относится
возможность работы с подписками на события. Подписка может осуществляться на
следующие события:
- Появление панели/скроллинга на экране.
- Удаление панели/скроллинга с экрана.
- Ответ на окно сообщения.
- Реакция на окно редактирования файла.
Если при воспроизведении скрипта возникло непредвиденное событие, то будет выполнен
пересмотр всех имеющихся подписок. Если найдена подписка, параметры которой
удовлетворяют данным возникшего непредвиденного события, то будет выполнена
заданная при создании подписки макрофункция.
Пример использования механизма подписок
При выполнении массовой процедуры выводится сообщение "Продолжить
выполнение операции?". На данное сообщение необходимо ответить
положительно. Реализация данной подписки может быть следующей:
macro Subscription_ContinueAsk()
  MsgWindow("Да","Продолжить выполнение операции?");
end;
CreateSubscription("MySubscriptionForContinue",
MsgWindow(undef,"Продолжить 
выполнение 
операции?"),
Subscription_ContinueAsk);
Для эффективного управления подписками при создании подписки указывается ее имя с
тем, чтобы в дальнейшем можно было подписку выключить/включить и удалить/заново
добавить. Указанные действия с подписками выполняются с помощью следующих команд:
- DeleteSubscription("имя подписки");
- EnableSubscription("имя подписки");
- DisableSubscription("имя подписки").
На время выполнения подписки все другие подписки выключаются.
Вывод данных в лог-файлы
При работе утилиты управления тестами RS-ATM формируются лог-файлы трех типов:
- Лог выполнения скрипта. Формируется в обязательном порядке при выполнении скрипта в
режиме воспроизведения.
- Лог возникших ошибок. Формируется в случае возникновения ошибок в ходе выполнения
скрипта. Содержит все ошибки, возникшие при воспроизведении скрипта.
- Информационный файл.
Для вывода данных в лог выполнения скрипта и информационный файл используются
следующие команда:
- Log(string)
- запись данных в файл выполнения скрипта.
- Inf(string)
- запись данных в информационный файл.
- LogAndInf(string[,boolean])
- запись данных и в файл выполнения скрипта, и в
информационный файл одновременно.
Команда LogAndInf имеет второй необязательный аргумент типа boolean: если он
установлен в true, то утилита управления тестами сымитирует ошибку выполнения скрипта
и, если не настроен Интеллектуальный выход в Монитор ИБС, воспроизведение будет
остановлено.
Интеллектуальный выход в Монитор ИБС
При работе утилиты тестирования RS-АТМ существует возможность перезапуска скрипта с
помощью включения режима интеллектуального выхода в Монитор ИБС.
При включенном режиме интеллектуального выхода в Монитор ИБС в случае возникновения
ошибки утилитой тестирования будет осуществлен выход в Монитор ИБС, эмулируя при
этом действия по выходу из панелей, скроллингов и автоматически отвечая на все вопросы
ИБС, например, "Запись была изменена. Сохранить?".
Режим интеллектуального выхода в Монитор ИБС не является до конца адаптивным,
поэтому существует возможность ограничивать утилиту тестирования по глубине вероятной
вложенности интерфейса. Например, если известно, что максимальная вложенность
интерфейса в процессе работы скрипта не будет превышать 4-х (уровни Подсистема,
Скроллинг, Вложенный скроллинг, Справочник), то ограничивать максимальный уровень
вложенности интерфейса необходимо числом 5 (на уровень больше вложенности
сообщения, которое и может стать сигналом к включению данного режима). В этом случае
утилитой тестирования будут выполнены следующие 5 действий эмуляции:
- Нажата клавиша [Esc], ответ на окно сообщения об ошибке.
- Нажата клавиша [Esc], выход из справочника.
- Нажата клавиша [Esc], выход из вложенного справочника.
- Нажата клавиша [Esc], выход из скроллинга.
- Выбор пункта меню 9999 и выход из подсистемы.
Максимальный 
уровень 
вложенности 
задается 
в 
системной 
переменной
АТМ_MAX_CONFIRMATIONS_DEPTH
. По умолчанию установлено ограничение в 20
уровней вложенности интерфейса.
Для того чтобы обеспечить корректную работу режима интеллектуального выхода в Монитор
ИБС, необходимо выполнить следующие действия:
- В системной переменной АТМ_INTELLI_EXIT_IF_ERROR
 установить значение "true".
- В системной переменной АТМ_PLAY_TIMEOUT
 установить значение, отличное от 0.
Например, если скрипт состоит из нескольких итераций, и на любой итерации возможно
появление непредвиденной ошибки, то в переменной целесообразно указывать значение,
близкое к общему времени выполнения всех итераций в секундах.
- Определить макрофункцию Main(), первой эмулируемой командой которой должна быть
PanelOn("Список",undef,undef,"L_MON2")
.
- Запустить режим воспроизведения скрипта, указав ключ /hide_msg.
Пример 
скрипта 
для 
корректной 
работы 
режима
интеллектуального выхода в Монитор ИБС
ATM_INTELLI_EXIT_IF_ERROR=true;
ATM_PLAY_TIMEOUT=5;
macro Main()
  PanelOn("Список",undef,undef,"L_MON2");
  ...
end;
Main();
Выключить режим интеллектуального выхода в Монитор ИБС можно в любом месте,
установив в переменной АТМ_PLAY_TIMEOUT
 значение, меньшее значения переменной
ATM_PLAY_TIMER
 или равное нулю. После того, как это сделано, при возникновении
ошибки воспроизведения утилита тестирования остановит воспроизведение, выйдет в
Монитор ИБС и нажав там клавишу [Esc] выполнит выход из ИБС.

Для эффективного взаимодействия утилиты тестирования с RSL-макросами в них можно

"welcome!"

  ATMSetValue("b","hello!");
end;
Как видно из примера, команда RunMacro вызывает определенную макрофункцию в
указанном макрофайле. Если указать только один параметр команды RunMacro, то
макрофайл будет только скомпилирован. Для этого случая он должен иметь следующий вид:
import "rsatm";
macro TestMacroFunction()
var
  my_a:integer, my_b:string;

  my_b=ATMGetValue("b"); // переменная my_b теперь равна "welcome!"

  ATMSetValue("b","hello!");

TestMacroFunction();

## Управление трассировкой

Внимание!
Информация данного раздела используется только при работе ИБС
RS-Bank V.6 на СУБД Oracle
В утилите тестирования RS-АТМ предусмотрена возможность включения и выключения
трассировки RS-Trace и ORACLE-трассировки. Для этого предназначены следующие
команды:
- EnableRSTrace();
- DisableRSTrace();
- EnableORATrace(string);
- DisableORATrace();
Для идентификации файла ORACLE-трассировки в команду EnableORATrace необходимо
передать аргумент типа String, текст которого будет дописан к файлу трассировки.

## Синхронизация

нескольких процессов
RS-ATM
Для управления одновременным воспроизведением скриптов при многопользовательском
тестировании в RS-АТМ используются специальные объекты – события синхронизации. Эти
события оповещают один поток о том, что другой поток выполнил какое-то действие.
Например, один скрипт завершил ввод клиентов, другой скрипт может приступать к вводу
счетов. Это необходимо, чтобы избежать ошибок при попытке нескольких скриптов получить
доступ к одним и тем же ресурсам.
События синхронизации задаются в строковом виде. Могут быть установлены, сброшены и
ожидать выполнения. Их можно использовать с помощью следующих команд:
- SetSyncEvent (string) – создает событие с именем, переданным в качестве параметра.
- WaitSyncEvent (string[, integer]) – ожидает наступления события. В первом параметре
содержится имя события. Второй параметр содержит время ожидания в секундах. Если
второй параметр задан, то ожидаем наступления события или истечения времени. Иначе,
если второй параметр не задан и событие не наступило, работа скрипта будет
прекращена.
- ResetSyncEvent (string) – сбрасывает событие. В параметре содержится имя события.
Пример использования событий синхронизации
Скрипт test1.atm:
SetSyncEvent("ВводКлиентов");
//Ввод клиентов завершен, наступило событие "ВводКлиентов"
WaitSyncEvent("ВводСчетов");
//Ждем наступления события "ВводСчетов"
ResetSyncEvent("ВводКлиентов"); 
// Сбросим событие "ВводКлиентов".
Скрипт test2.atm:
WaitSyncEvent("ВводКлиентов", 30);
// Ждем наступления события "ВводКлиентов" 30 секунд
SetSyncEvent("ВводСчетов");
// Ввод счетов завершен, наступило событие "ВводСчетов"

Утилита управления тестами RS-ATM
Утилита управления тестами RS-ATM (RS-Application Testing Machine) предназначена для
обнаружения ошибок в ИБС RS-Bank V.6 и позволяет клиентам компании R-Style Softlab
самостоятельно выполнять тестирование системы. Суть работы утилиты заключается в
последовательном выполнении следующих действий:
- Воспроизведение ранее созданного с помощью инструмента записи и воспроизведения
событий RS-ATM файла скрипта.
- Сравнение результата выполнения исходного файла скрипта с результатом выполнения
файла скрипта, записанного при повторном воспроизведении исходного файла.
