## Загрузка оперативных данных

Загрузка оперативных данных из файла сессии, находящегося в каталоге ..\IMPORT,
осуществляется в следующем порядке:
· Разархивируются сессии.
· Открываются таблицы базы данных.
· Выбирается загружаемая сессия.
· Проверяется соответствие электронной цифровой подписи.
· Создается обработчик корневого тега и объект-парсер, в конструктор которого передается
созданный обработчик.
· Регистрируется в базе данных факт завершения загрузки сессии.
Репликация данных в формате XML
· Закрываются таблицы базы данных.
При загрузке одной записи выполняются следующие действия (приведен процесс
формирования буфера записи):
· Для каждого значения поля записи:
· Определяется имя и тип следующего поля в структуре, полученной из файла сессии.
· Определяется тип и смещение поля с таким именем в записи реальной таблицы (по
структуре из словаря).
· Если типы данных различаются, то выполняется конвертация данных в тип поля
реальной таблицы.
· Данные помещаются в буфер записи по необходимому смещению.
· Если в файле содержится тег ovl
, описывающий оверлей постоянной части записи, то
определяется смещение оверлея в буфере записи таблицы и данные из тела тега
загружаются в буфер записи так же, как поля собственно структуры записи. Определяется
по атрибуту оверлея последнее "накрываемое" им поле и выполняется переход на
следующее за ним.
· Переменная часть записи, заданная тегом ovl
, загружается аналогично оверлею
постоянной части записи. Размер переменной части в этом случае определяется по
размеру ее структуры в словаре.
· Переменная часть, заданная тегом bvp
, формируется на основе шестнадцатеричного
представления ее байтов. Размер определяется атрибутом size тега bvp
.

## Репликация справочников

Как правило, в работе банка возникают ситуации, когда требуется изменять технологию
работы. Обычно это связано с изменениями проведения тех или иных банковских операций.
Поэтому любая интегрированная банковская система должна обладать средствами
оперативного реагирования на изменяющиеся условия проведения операций.
Примером такого подхода в АС RS-Retail V.6 является возможность внесения информации о
новых видах банковских продуктов и сопутствующих им атрибутов, существенно
отличающихся от существующих, в справочники системы.
Вся информация об изменениях в видах банковских продуктов, их настройках и другие
сведения, содержащиеся в справочниках системы и базе счетов, пересылаются из бэк-
офиса во фронт-офисные подразделения с помощью файлов репликации, которые по
умолчанию содержатся в директории ..\TRAKFILE.
Репликация справочной информации и базы счетов осуществляется в подсистеме "Сервис
розничных услуг" (см. раздел "Передача данных, содержащихся в справочниках системы"

## Выгрузка справочников

Выгрузка данных осуществляется следующим образом:
· Открываются таблицы базы данных.
· Начинается корневой тег objreplication

· Записывается описание файла (тег description

· Записывается информация об источнике (тег source

· Записываются собственно объекты (тег object

· Завершается корневой тег objreplication

· Сформированный файл подписывается электронной цифровой подписью и архивируется.
· Закрываются таблицы базы данных.
Выгрузка объекта выполняется по следующему алгоритму:
· Общий механизм работы с атрибутами (т.е. единый для репликации оперативных данных
и справочников) предусматривает следующие этапы:
· Если выполняется первая выгрузка объекта определенного типа в текущем сеансе
репликации, то определяется структура записи, все возможные структуры переменных

.
· Определяется структура записи, структура переменной части и набор структур оверлеев
для текущей записи
.
· Начинается тег object
.
· При необходимости в файл записываются метаданные (тег metadata
). Общий
механизм работы с атрибутами подразумевает использование метаданных, в частности,
если запись, представляющая объект в базе данных, имеет переменную часть или
оверлеи. При этом для каждого оверлея создается метаданное (тег meta
) с
name=overlay и value=<имя структуры в словаре>.
· Записываются атрибуты (тег attributes
). 
· Записываются подчиненные объекты и списки объектов (тег dependent
).
· Завершается тег object
.
При выгрузке списка объектов в рамках тега objlist

После включения профилировщика начинается сбор информации, т.е. подсчет количества
операций обращения к базе данных и времени их выполнения.

следует убедиться в отсутствии в таблице данных с таким же значением поля t_cnum.
Внимание!
Данные выгружаются в таблицу только при корректном выключении
