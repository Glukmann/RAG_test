# Reports InstrExp

## Введение

Инструмент Windows Reports предназначен для реализации новых или перевода
имеющихся txt-отчетов в форматы наиболее популярных приложений офисного пакета, а
также форматы браузера: XLS, RTF, DOC, HTML.
Благодаря представленному инструментарию, пользователям, наряду с формированием
отчета в привычном txt-формате, становится доступен обширный функционал этих
программных средств, а именно:
- создание отчетных документов в соответствии с предварительно заданными размерами;
- формирование

данных

- автоматическое суммирование значений по группам однородных данных, вывод
промежуточных итогов;
- сортировка данных в отчетах, возможность изменять порядок сортировки (в табличных
отчетах);
- построение диаграмм на основе табличных данных;
- создание многозакладочных отчетов (в XLS-формате);
- автоматическая группировка строк по диапазону столбцов;
- ручное и автоматическое масштабирование;
- автоматический подбор ширины поля для вывода информации и т.д.
На сегодняшний день инструмент Windows Reports предоставляет широкие возможности,

Инструмент

Ключевой объект Windows Reports – CMakeReport – предлагает пользователям богатый
выбор возможностей, которые условно подразделяются на три функциональные группы:
- первая используется для выпуска отчета в текстовом формате TXT;
- вторая предназначена для отчетов в формате одного из приложений офисного пакета;
- третья позволяет использовать оба эти варианта.
Большинство разработчиков, занимающихся разработкой отчетов на RSL, регулярно
сталкивается с рутинными задачами по выводу уже полученной информации на печать.
Многие

не

информации на печати, что приводит к необоснованному увеличению объема макросов,
тиражированию незамеченных ошибок и отсутствию унификации в создаваемых отчетах. 
Объект CMakeReport в первую очередь служит для устранения указанных недостатков, а
также для значительного упрощения создания отчетов и возможности их быстрого переноса
из консольного представления в современные виды отображения. С помощью этого объекта
можно без затруднений получать самые разнообразные отчеты, как содержащие таблицы,
так и не содержащие. Наиболее простое решение реализовано для отчетов типа "Строки
заголовка" + "Таблица" + "Строки подвала".
В настоящий момент объект CMakeReport позволяет выводить отчеты как в формате TXT,
так и в формате HTML, причем как в двухуровневой архитектуре, так и в трехуровневой, при
этом исходный код для отчетов является единым. Получающиеся HTML-файлы отчетов
выводятся в xls-формате, но в большинстве своем легко читаются также и приложениями

Кроме того, существует возможность получать отчеты в других широко распространенных
форматах: XLS, DOC, RTF.

В текстовом варианте:
- Самостоятельно отслеживает высоту страницы и достаточно корректно осуществляет
переход к следующей странице.
- При необходимости раздвигает по высоте непомещающиеся строки отчета, в том числе

- Для строк таблицы отчета автоматически формирует все необходимые табличные

пользовательские символы.
- Для таблиц автоматически переносит заголовок на следующую страницу. Если таблица
начинается на одной странице, но на эту страницу не помещается хотя бы одна строка

заголовок.
- При любом начале строк таблицы автоматически выводит шапку таблицы, а при
окончании строк таблицы (т.е. перед любой простой строкой) корректно ее завершает.
- Позволяет иметь в таблице элементы разной ширины, при этом все сочленения между
ними таблицы будут корректными.
- Позволяет настраивать наличие межстрочных разделителей, как для всей таблицы, так и
для отдельных ее строк.

Принцип работы
Главным преимуществом объекта CMakeReport является то, что он позволяет разделить
между собой данные и их представление. Задача разработчика отчета сформировать
данные для объекта CMakeReport, а сам процесс вывода на печать объект берет на себя.
Данные в объекте хранятся в виде массива "строк". Каждая строка представляет собой
массив ячеек с информацией.
Ячейка является экземпляром объекта CParamPrintCell и может содержать как данные для
простой строки отчета, так и данные для табличной строки отчета. Необходимо отметить,
что каждая строка должна содержать хотя бы одну ячейку, иначе при добавлении, она
просто не будет добавлена в объект. Тип строки определяется по типу первой ячейки
строки. По умолчанию каждая ячейка считается ячейкой таблицы отчета.
В каждой ячейке отчета хранится следующая информация:
- Тип (в соответствии с основными типами данных, применяемых в RSL).
- Количество символов в ширину, на которое должно быть напечатано значение (для
значения строкового типа, при недостатке ширины, происходит раздвижка печатной строки
по вертикали).
- Количество знаков после запятой (анализируется объектом CMakeReport только для
дробных чисел и денег).
- Дополнительные модификаторы в виде строки символов, разделенных двоеточием, и
модификаторы расширенного форматирования.
- Вид представления элемента, в соответствии с константами представления:
- REP_ELEM_TABL – строка таблицы.
- REP_ELEM_STR – простая строка.
- REP_ELEM_SHOWSEP – строка таблицы с обязательным разделителем строк перед
ней.
Перед выводом данных в отчет в формате офисных приложений на сервере создается
временный файл в формате .htm, содержащий всю необходимую информацию для
построения отчета в указанном формате. Для многозакладочных отчетов создается также
временная директория, в которой содержатся временные файлы, соответствующие
закладкам отчета. В процессе печати отчета все данные будут перенесены в офисное

Имя временного файла формируется следующим образом:
<дата и время с миллисекундами> + <номер коннекта пользователя>.htm
Например, 23030914444223_001.htm.
Имя временной директории имеет следующий формат:
<дата и время с миллисекундами> + <номер коннекта пользователя>.files
Например, 23030914444223_001.files.
Готовый отчет записывается в файл, имя которого формируется следующим образом: 
имя выходного потока_<номер коннекта>.<расширение соответствующего формата>
Например, t_list_001xls. 
Есть возможность установить пользовательское имя файла. Это можно сделать с помощью
методов: 
SetFileName(Name:string), 
SetFileNameWin(Name:string),
SetFileNameTxt(Name:string), которые устанавливают имена выходного файла для отчетов
в форматах офисных приложений и текстового отчета. В этом случае к имени Name будет
добавлен номер коннекта и соответствующее формату расширение.
Если файл отчета с аналогичным именем захвачен соответствующим приложением
офисного пакета, то к имени файла будет добавлен уникальный код <дата и время с
миллисекундами>.
Файловый состав
В состав инструмента входят группа обязательных и группа вспомогательных файлов:
- Обязательные файлы – необходимые классы, настройки, константы и утилиты, без
которых функционирование инструмента НЕВОЗМОЖНО.
- Вспомогательные файлы – дополнительные унифицированные возможности для
функционирования инструмента. Эти возможности являются автономными, т.е. для их
использования надо специально импортировать их в прикладные макросы.
Такой подход будем в дальнейшем именовать "дружественными классами и функциями".
Основной их особенностью является то, что для их функционирования в качестве одного из
параметров передается ссылка на уже созданный объект класса CMakeReport.
К основным файлам относятся:
- or_class.mac – содержит все вспомогательные классы для инструмента;
- or_const.mac – содержит все глобальные константы для инструмента;
- or_dps_h.mac – отвечает за работу с диапазонами, заданными строкой;
- or_exl_h.mac – отвечает за работу с приложениями офисного пакета, а также содержит
текущую версию инструмента и список обязательных файлов;
- or_hdr_h.mac – содержит "конструктор заголовков";
- or_rep_h.mac – содержит главный класс CMakeReport;
- or_set_h.mac – содержит все глобальные настройки для инструмента;
- or_srt_h.mac – отвечает за сортировку;
- or_tools.mac – содержит все "низкоуровневые" функции для инструмента;
- or_worm.mac – используется для самостоятельного размножения инструмента на
вызвавшие его терминалы;
- or_trm_h.mac – отвечает за работу инструмента на терминале;
- trlocale.d32 – специальная DLM для работы с локальными настройками Windows на
терминале;
- fileop.d32 – специальная DLM для работы с файловой системой на терминале.

Методы
Основным интерфейсом для разработчика отчетов, использующего объект CMakeReport,
конечно, являются методы этого объекта. Класс CMakeReport включает в себя набор
внутренних и пользовательских методов. Пользовательские методы будут рассмотрены
достаточно подробно, а о внутренних методах будет только краткое упоминание.
Использование внутренних методов ЗАПРЕЩЕНО, хотя RSL и позволяет обращаться к ним
напрямую.
Все методы разделены на 4 условные группы.
- Методы ядра
- служат основой для всех действий с объектом и позволяют получить
любой текстовый отчет. Их изменение (например: переопределение в наследнике класса
CMakeReport) требует особой аккуратности и понимания принципов построения и работы
объекта.
- Утилитные методы
- служат для облегчения работы с объектом и выполнения
некоторых стандартных вспомогательных действий (например, поиск, получение или
задание некоторых параметров и т.д.).
- Методы для Windows-отчетов
- служат основой для получения Windows-отчетов. Их
изменение также требует особой аккуратности и понимания принципов построения и
работы объекта, но, кроме того, требует элементарных знаний по ActiveX-объектам
стандартных офисных приложений Excel и Word, а также хотя бы минимальные знания по
HTML.

для использования класса CDiapason, формирующего массив столбцов

которой каждая новая строка не сразу добавляется в конец отчета, а сначала по
ней формируется индекс автогруппировки, затем он сравнивается с индексами
предыдущих введенных строк. Если находится строка с одинаковым значением
индексов, то она заменяется новой строкой. Но при этом все ячейки этих строк
типа Double и Money складываются. При незаполненном массиве столбцов
автогруппировки группировка не производится. По умолчанию группировки нет.
Внимание!
Следует иметь в виду, что для больших отчетов автогруппировка
может сильно замедлить создание отчета, так как пока там
используется простая процедура перебора вышестоящих строк,
начиная с последней, до тех пор, пока не найдется позиция новой
строки.

Одним из важнейших достоинств объекта CMakeReport является возможность очень гибко

Отвечает за стиль шрифта (FS – Font Style). Имеет следующие допустимые аргументы:
B, b – жирный шрифт (Bold).

I, i – курсивный шрифт (Italic).
U, u – подчеркнутый шрифт (Underline).
T, t – верхний индекс (Top).

Формирование отчетов в MS Excel в
трехуровневой архитектуре
При работе в трехуровневой архитектуре возможны два варианта в зависимости от
настройки FlagTerminalClone:
- Если настройка имеет значение true (значение по умолчанию), файлы инструмента
достаточно иметь только на сервере. При запуске макроса отчета, реализованного на
инструменте Windows Reports, происходит автоматический поиск терминальной части
инструмента на терминале в папке макросов, заданной в модуле or_set_h.mac переменной
NameMacDirTerm (по умолчанию папке присвоено имя "$Mac\\"):
- Если инструмент на терминале не найден, то происходит копирование следующих
файлов, заданных в первых строках модуля or_exl_h.mac с сервера на терминал:
- or_trm_h.mac;
- or_exl_h.mac;
- or_const.mac;
- trlocale.d32;
- fileop.d32.
- Если инструмент найден, то производится сравнение версий (версия инструмента
задается во второй строке модуля or_exl_h.mac).
В случае если версии инструмента на сервере и терминале различаются, то происходит
автоматическое копирование терминальной части инструмента на терминал. Если версии
совпадают, то копирование не выполняется.
- Если настройка имеет значение false, файлы инструмента необходимо самостоятельно
переносить на все нужные терминалы. При обновлениях это придется делать каждый раз
заново, так как версии инструмента должны совпадать как на сервере, так и на терминале
для корректной работы инструмента.
Перед работой в настройках инструмента должны быть точно прописаны пути (лучше всего
относительные) к серверу и терминалу.

Автоподбор ширины выводимой
информации
Если отчет содержит таблицу, то не нужно указывать ширину данных в таблице или
указывать ее нулевой, так как по умолчанию ширина берется по ширине столбцов заголовка
таблицы. Это позволяет при необходимости изменять ширину любого столбца простым
перерисовыванием заголовка, не трогая код. Если отчет не содержит таблицу, то ширину
выводимой информации лучше указывать не в абсолютных единицах, а в относительных,
чтобы опять же можно было легко менять поля для вывода данных изменением одной
константы (например, константы отвечающей за максимальную ширину отчета). Но самым
простым способом является использование метода AutoScan, который позволяет задавать
данные по принципу квадратных скобок в классическом RSL.

Автоинкрементирование в столбце
Часто в прикладных макросах необходимо нумеровать строки таблицы. Чтобы не заводить
там специальных переменных, в которых ведется подсчет введенных строк, можно
использовать специальное значение для ячейки таблицы: KEY_AUTOINC. Каждый раз, когда
в новой строке появляется такое значение, вместо него выводится значение встроенной
автоинкрементной переменной. Эта возможность особенно полезна при использовании
автогруппировки, так как автоматически учитывается, добавилась ли новая строка в отчет
или модифицировалась одна из уже введенных строк.
Пример фрагмента кода, в котором в первом столбце будет производиться нумерация строк
таблицы:
…

Rep.AddPrintCell(KEY_AUTOINC);
Rep.AddPrintCell(«Второй столбец»);
Rep.AddPrintCell(«Третий столбец»);
I = I + 1;
End;
…
Использование встроенной сортировки
строк
Встроенная сортировка сформированных строк отчета – еще одна очень удобная сервисная
возможность класса CMakeReport. В некоторых случаях она бывает просто необходима,
например, если в базах, по которым строится прикладной отчет, нет необходимых ключей
сортировки. 
Воспользоваться сортировкой достаточно просто. Для этого вызываем нужное количество
раз метод AddSortField(_Number:integer, _Direct:integer), где:
Number – номер колонки сортировки.
Direct – направление (> 0 – по возрастанию, < 0 – по убыванию). Приоритет сортировки
столбцов соответствует порядку вызовов этой функции. Если до этого уже
осуществлялась сортировка, то для начала новой сортировки необходимо
очистить поля массив полей сортировки вызовом метода ClearAllSortField().
После получения записей отчета, для непосредственно процесса сортировки, нужно
воспользоваться методом SortRep(_BegDiap:integer, _EndDiap:integer), где BegDiap,
EndDiap – границы диапазона сортировки.
Следует отметить, что для получения границы диапазона очень удобно два раза
воспользоваться методом GetNumLastStr(): перед началом таблицы BegDiapTab =
GetNumLastStr() и сразу после окончания таблицы EndBegDiapTab = GetNumLastStr() – 1.
Следует помнить, что GetNumLastStr() возвращает номер строки, в которой будет
производиться запись при очередном вызове метода AddStr(). Проще говоря, возвращается
размер массива уже введенных строк.

Встроенная возможность вывода подытогов – одна из полезных сервисных возможностей
класса CMakeReport. Ею можно пользоваться одновременно с сортировкой.
Воспользоваться возможностью вывода подытогов также просто, как и сортировкой. Для
этого вызываем необходимое количество раз метод AddSortField(_Number:integer,
_Direct:integer, 
_NumColSubRes:integer, 
_TextSubRes:string,
_CountStrForShowSubRes:integer), где:
Number – номер колонки сортировки.
Direct – направление (> 0 – по возрастанию, < 0 – по убыванию).
_NumColSubRes – номер колонки, в которую будет выведена строка, переданная в
параметре _TextSubRes (по умолчанию номер колонки – 0 и текст "ИТОГО").
_CountStrForShowSubRes – минимально необходимое число строк для формирования
строки подытогов (по умолчанию – 1, т.е. все подытоги).
Самое главное – перед подведением подытогов должен быть вызван метод SortRep(). При
этом массив полей сортировки может содержать не только поля для подведения подытогов,
но и поля для сортировки. Например, можно отсортировать некую базу по подразделениям,
сотрудникам и начислениям, а подвезти подытоги только по сотрудникам. Приоритет
подведения подытогов, также как и при сортировке, соответствует порядку вызовов метода
AddSortField(). Если до этого уже осуществлялась сортировка, то для начала новой
сортировки необходимо очистить поля массив полей сортировки вызовом метода
ClearAllSortField().
После получения записей отчета для непосредственно процесса сортировки необходимо
воспользоваться 
методом 
CreateSubRes(_BegDiap:integer, 
_EndDiap:integer), 
где
BegDiap, EndDiap – соответствующие границы диапазона подведения подытогов.
Следует отметить, что для получения границы диапазона удобно дважды воспользоваться
методом GetNumLastStr(): перед началом таблицы BegDiapTab = GetNumLastStr() и сразу
после окончания таблицы EndBegDiapTab = GetNumLastStr() – 1. Напомним, что

GetNumLastStr() возвращает номер строки, в которой будет производиться запись при
очередном вызове метода AddStr(). Проще говоря, возвращается размер массива уже
введенных строк.
Кроме того, если таблица была пронумерована, то перед вызовом метода CreateSubRes()
очень 
удобно 
вызвать 
метод 
RenumRep(_NumCol:integer, 
_BegDiap:integer,
_EndDiap:integer), где:
_NumCol – номер колонки, в которой необходимо выполнить перенумерацию.
Пример.
Фрагмент кода, в котором производится сортировка, перенумерация
отчета и подведение подытогов:
/* Запомним номер первой строки таблицы */ 
NumStr = Rep.GetNumLastStr();
…
/* Подводим подытоги по 3 и 6 столбцу, строки о подытогах выводим в
пятый столбец */
Rep.AddSortField(3,  1, 5, "Итого по предприятию");
Rep.AddSortField(6, 1, 5, "Итого по отделу");
/* Для 9 столбца будем только сортировать, без подведения подытогов */
Rep.AddSortField(9,  1);
/* Произведем сортировку и перенумерацию отсортированных строк */
Rep.SortRep(NumStr, Rep.GetNumLastStr() - 1);
Rep.RenumRep(0, NumStr, Rep.GetNumLastStr() - 1);
/* Произведем подведение подытогов */
Rep.CreateSubRes(NumStr, Rep.GetNumLastStr() - 1);
Скрытие столбцов
Скрытие столбцов – уникальная возможность минимальным количеством кода добиться
большого разнообразия получаемых вариантов прикладного отчета. Например, если вывод
каких-то столбцов зависит от флажков в панели настроек, то ненужно плодить огромного
количества ветвящегося кода, а достаточно получать общий отчет, в котором просто
скрывать ненужные столбцы вызовом одной функции.
Другая очень полезная возможность, которую можно использовать вместе со встроенной
сортировкой – навигационные столбцы. Т.е. если прикладной отчет нужно отсортировать по
одному или нескольким параметрам, но эти параметры не попадают в отчет, то можно
сделать следующее: добавить необходимое количество столбцов с параметрами в таблицу,
сформировать строки отчета, отсортировать по этим дополнительным столбцам, а затем
столбцы с параметрами просто скрыть.

Автосуммирование в столбце
Часто в прикладных макросах необходимо получать итоги по столбцу таблицы. Чтобы не
заводить там специальных переменных, в которых ведется подсчет необходимых сумм,
можно использовать специальное значение для ячейки таблицы: KEY_SUMABOVE. Каждый
раз, когда в новой строке появляется такое значение, вместо него выводится значение всех
значений таблицы, находящихся выше данной ячейки.

Для того чтобы табличный отчет, выпускаемый с помощью Windows Reports в текстовом

TEMPLATE_DEFAULT = 0}, состоящий из последней строки шапки таблицы. Для основной

нужно отделить шаблоном по умолчанию {AddStr(FALSE, TEMPLATE_DEFAULT)}.

SetTemplate(TEMPLATE_1, «│    │      │       │     │   ├────────┼────
────────┼────────────┼────────────┼────────────┤»);
а затем последовательно вызывать добавления строк, с шаблоном по умолчанию и с
шаблоном с номером TEMPLATE_1.
Пример.
While()
    /* Заполнение ячеек первой строки */
    ObjPrint.AddStr(FALSE, TEMPLATE_DEFAULT); // Добавление первой
строки с шаблоном по умолчанию
    /* Заполнение ячеек второй строки */
    ObjPrint.AddStr(FALSE, TEMPLATE_1); // Добавление второй строки с
шаблоном по TEMPLATE_1
End;

Размер отчета в формате MS Excel
В конверторе MS Excel из html-файла есть ограничение, которое заключается в том, что
конвертор не воспринимает объединений более чем int16 ячеек, т.е. 32763 ячеек на одном
листе.
При этом в целях корректного форматирования ячеек отчета необходимо следить за тем,
чтобы ячейки основной таблицы состояли из одной колонки без объединения. Для
выполнения этого условия шапка отчета не должна разбивать отчет на мелкие колонки, т.е.
должна состоять из одной колонки.

Количество полей в документе слияния
Технология документов слияния имеет ограничение на количество полей в шаблоне, а

Для выхода из этой ситуации "большой" шаблон можно разбить на отдельные шаблоны.

оборотной стороны.

Технология работы с xls-шаблонами
Технология работы с инструментарием по выпуску отчета на основании xls-шаблона
предусматривает определенную последовательность действий.
- Подготовка шаблона
Подготовка шаблона проводится в три этапа:
- Присвоение имен ячейкам или диапазонам, в которые необходимо занести данные
(см. Рис. 4
).
Рис. 4. Присвоение имен ячейкам или диапазонам.
- Присвоение имени табличной шаблонной строке и каждой ячейке табличной строки
(см. Рис. 5
). Табличная строка и ячейка табличной строки могут быть любой
сложности и занимать любое количество строк и столбцов xls-файла. Например, в
качестве строки таблицы может быть задан диапазон A7:C8, а ячейки таблицы могут
быть сформированы слиянием двух ячеек, допустим, A7 и A8 (см. Рис. 5
).
Рис. 5. Присвоение имени табличной шаблонной строке и каждой ячейке табличной строки.
- Присвоение имени строкам шаблонам подытогов (см. Рис. 6
). Присвоение имени
строкам шаблонам подытогов, а также их ячейкам выполняется аналогично табличной
строке. Шаблонов подытогов может быть столько, сколько это нужно для решения
конкретной прикладной задачи. Например, могут быть два вида подытогов одной
таблицы: "Итого за месяц" и "Итого за год" (см. Рис. 6
).
Рис. 6. Присвоение имени строкам шаблонам подытогов.
- Заполнение шаблона
Заполнение шаблона происходит в макрокоде.
Процедура заполнения состоит из следующих этапов:
- Создание объекта CTemplateXLS.
var ObjTempl:CTemplateXLS = CTemplateXLS();
- Открытие шаблона методом ObjTempl.OpenTemplate("Шаблон.xls").
- Заполнение значениями именованных полей с помощью метода SetValue_NameCell().
ObjTempl.SetValue_NameCell("ФИО", "Юркова Татьяна Александровна")
- Регистрация шаблона таблицы, с помощью метода ObjTempl.RegisterTable(). Метод
возвращает объект "таблица" (типа CTable).
Table = ObjTempl.RegisterTable("Table")
- При необходимости можно зарегистрировать строки шаблонов подытогов, с помощью
метода Table.RegisterSubItog() объекта "таблица", который был получен при
регистрации.
Table.RegisterSubItog("SubTitle" , SUBTITLE_1)
- Заполнение значениями ячеек шаблона строки таблицы, с помощью метода
SetValueCell объекта "таблица", который был получен при регистрации.
Table.SetValueCell("Cell1", MonName(i))
- Добавление строки в таблицу, с помощью метода AddStr() объекта "таблица".
- Добавление строки подытогов, с помощью метода Table.AddStrSubItog() объекта
"таблица".
- Завершение таблицы. После добавления всех строк таблицы необходимо вызвать
метод EndTable() объекта "таблица", который удалит шаблонные строки таблиц и
подытогов.
- Сохранение отчета с помощью объекта SaveAsTemplate(). Если в эту функцию будет
передан строковый параметр, то под этим именем и будет сохранен полученный отчет.
Если же параметр передан не будет, то отчет будет сохранен по общим правилам
.
ObjTempl.SaveAsTemplate()

При работе с шаблонами и csv-файлами следует помнить, что если в шаблоне строки

колонок занимает строка таблицы, столько же ячеек с данными и должно быть в csv-файле.

Не рекомендуется использовать в качестве расширения файла данных

настроек, передаваемых ему при открытии файла данных (например,
разделитель десятичных, разделитель тысяч, кодировка и т.п.).

Общие вопросы
Вопрос
У меня при запуске отчета выдается сообщение: "Ошибка выполнения Expression
syntax". Почему это бывает?
Ответ
Эта ошибка связана с неправильно построенным синтаксисом выражения для функции
ExecExp. Для точного определения причин ошибки необходимо сразу же после создания
объекта CMakeReport вызвать метод SetFlagDebug(true). Это позволит увидеть в
текстовом отчете все вызовы функции ExecExp и узнать причину ошибки выполнения по
последней распечатанной строке.
